## Sale API

Track sales by sending a `POST` request from your backend whenever a payment completes. Works with any payment provider — Stripe, Paddle, LemonSqueezy, or your own checkout.

See the [3-Step Integration Guide — Track Sales](/advertisers/tracking/quickstart-integration/#step-3--track-sales) for the full API reference including all parameters, response format, and error codes.

**Quick example:**

```js
await fetch('https://api.affitor.com/api/v1/track/sale', {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${process.env.AFFITOR_API_KEY}`,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    transaction_id: order.id,
    customer_key: currentUser.id,
    amount_cents: order.totalCents,
    currency: 'USD',
  }),
});
```

:::note
`transaction_id` must be unique per sale — Affitor uses it for deduplication. At least one of `customer_key` or `click_id` is required for attribution.
:::

## Bill Flow (Stripe)

Add Affitor metadata to your existing Stripe Checkout sessions. You collect payment normally; Affitor tracks via webhooks and invoices you weekly.

### Implementation

Include these metadata fields when creating Stripe Checkout:

#### One-Time Payment:

```javascript
// Helper to read Affitor cookies
function getCookie(name) {
  const value = `; ${document.cookie}`;
  const parts = value.split(`; ${name}=`);
  if (parts.length === 2) return parts.pop().split(';').shift();
  return null;
}

const clickId = getCookie('affitor_click_id');

const session = await stripe.checkout.sessions.create({
  line_items: [{
    price: 'price_xxx',
    quantity: 1,
  }],
  mode: 'payment',
  success_url: 'https://yoursite.com/success',
  cancel_url: 'https://yoursite.com/cancel',

  // ✅ REQUIRED: All three tracking fields
  metadata: {
    affitor_click_id: clickId,
    affitor_customer_key: currentUser.id,
    program_id: 'YOUR_PROGRAM_ID'
  }
});
```

#### Subscription:

```javascript
// Helper to read Affitor cookies
function getCookie(name) {
  const value = `; ${document.cookie}`;
  const parts = value.split(`; ${name}=`);
  if (parts.length === 2) return parts.pop().split(';').shift();
  return null;
}

const clickId = getCookie('affitor_click_id');

const session = await stripe.checkout.sessions.create({
  line_items: [{
    price: 'price_xxx',
    quantity: 1,
  }],
  mode: 'subscription',
  success_url: 'https://yoursite.com/success',
  cancel_url: 'https://yoursite.com/cancel',

  // ✅ REQUIRED: Session metadata (first payment)
  metadata: {
    affitor_click_id: clickId,
    affitor_customer_key: currentUser.id,
    program_id: 'YOUR_PROGRAM_ID'
  },

  // ✅ REQUIRED: Subscription metadata (recurring payments)
  subscription_data: {
    metadata: {
      affitor_click_id: clickId,
      affitor_customer_key: currentUser.id,
      program_id: 'YOUR_PROGRAM_ID'
    }
  }
});
```

### Server-Side Example (Node.js)

```javascript
// In your checkout endpoint
app.post('/api/create-checkout', async (req, res) => {
  // Extract affitor_click_id from cookies (sent in request headers)
  const cookies = req.headers.cookie || '';
  const clickId = cookies.split('; ')
    .find(row => row.startsWith('affitor_click_id='))
    ?.split('=')[1];

  // Get user from your authentication system
  const customerKey = req.user.id;

  const session = await stripe.checkout.sessions.create({
    line_items: [{ price: 'price_xxx', quantity: 1 }],
    mode: 'payment',
    success_url: 'https://yoursite.com/success',
    cancel_url: 'https://yoursite.com/cancel',

    // ✅ REQUIRED: All three tracking fields
    metadata: {
      affitor_click_id: clickId,
      affitor_customer_key: customerKey,
      program_id: process.env.AFFITOR_PROGRAM_ID
    }
  });
  
  res.json({ url: session.url });
});
```

:::tip[Required Tracking Fields]
Include all three fields (`affitor_click_id`, `affitor_customer_key`, `program_id`) in Stripe metadata for reliable payment attribution. The backend uses multiple lookup methods to ensure tracking works even if cookies are blocked.
:::
### How It Works

1. Customer clicks affiliate link → tracked with internal cookies
2. Customer signs up → calls `signup(customerKey, email)` with your internal user ID
3. Customer clicks buy → your checkout with `affitor_customer_key` in metadata
4. Customer completes payment via your Stripe
5. Stripe webhook fires → Affitor receives event
6. Affitor looks up customer-partner using `affitor_customer_key` → attributes to partner
7. Commission calculated → Affitor invoices you weekly

**Pros:** Simple integration, use your existing checkout, no cookie reading required  
**Cons:** Requires metadata setup, weekly invoice payment

## Attribution Lookup Strategy

When Affitor receives a payment webhook from Stripe, it uses a **4-step fallback mechanism** to find the customer-partner relationship and attribute the sale correctly.

### The Lookup Flow

```
Payment Event Received (Stripe webhook)
         ↓
Step 1: Lookup by click_id (from cookie)
         ↓
    Found? ──Yes──→ ✅ Link to attribution
         ↓
        No (cookies deleted)
         ↓
Step 2: Lookup by email
         ↓
    Found? ──Yes──→ ✅ Link + Update stripe_customer_id
         │          📍 Last-click attribution calculated from here
         │          (CPL/CPS tracking from lead/signup event)
         ↓
        No
         ↓
Step 3: Lookup by stripe_customer_id (returning customer)
        OR email_hash (from Stripe email)
         ↓
    Found? ──Yes──→ ✅ Link to attribution
         ↓
        No
         ↓
Step 4: Lookup by affitor_customer_key (if provided)
         ↓
    Found? ──Yes──→ ✅ Link to attribution
         ↓
        No
         ↓
❌ No attribution found (organic sale)
   → Payment NOT tracked (discarded)
```

### Why Multiple Lookup Steps?

| Scenario | Lookup Method | Example |
|----------|---------------|---------|
| Normal flow | `click_id` (Step 1) | Customer has cookies intact |
| Cookies cleared | `email` (Step 2) | Customer cleared browser data |
| Returning customer | `stripe_customer_id` (Step 3) | Customer made previous purchase |
| Cross-device | `email_hash` (Step 3) | Customer clicked on mobile, bought on desktop |
| Custom tracking | `affitor_customer_key` (Step 4) | You provided affitor_customer_key in metadata |

### What Affitor Receives from Stripe

When the webhook fires, Affitor extracts:

- Payment confirmation
- `program_id` from metadata
- `affitor_click_id` from metadata (if available)
- `affitor_customer_key` from metadata (if available, falls back to `user_id`)
- Customer email from Stripe
- `stripe_customer_id` from Stripe

### After Successful Lookup

Once a customer-partner relationship is found:

1. Verifies attribution is within 60-day window
2. Creates commission record
3. Updates `stripe_customer_id` if not already set (Step 2)

### Organic Sales

If no attribution is found after all 4 steps, the payment is considered an **organic sale** and is **NOT tracked** (discarded). This ensures only affiliate-referred sales generate commissions.

## Metadata Reference

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `affitor_click_id` | string | **Yes** | Click identifier from Affitor cookie (set automatically when user clicks affiliate link) |
| `affitor_customer_key` | string | **Yes** | Your internal user/customer ID (must match the `customerKey` from `signup()`). `user_id` is accepted as a backward-compatible alias. |
| `program_id` | string | **Yes** | Your Affitor program ID |

**All three fields are required for reliable payment attribution.**

:::caution[Required Metadata Fields]
All three metadata fields must be included:
- `affitor_click_id` - Read from Affitor cookie (primary lookup method)
- `affitor_customer_key` - Your internal user/customer ID (fallback lookup + validation). `user_id` is accepted as a backward-compatible alias.
- `program_id` - Your Affitor program ID (program identification)

Without these fields, the purchase cannot be attributed to the affiliate partner.
:::

### Important Notes

- **Click ID:** Set automatically by Affitor tracker when user clicks affiliate link. Read from cookie using the helper function.
- **Customer Key Must Match:** The `affitor_customer_key` must exactly match the `customerKey` you passed during `signup()`.
- **Program ID:** Find your program ID in the Affitor dashboard under Settings.
- **Hybrid Attribution:** Backend tries `affitor_click_id` first (primary), falls back to `affitor_customer_key` if cookies are blocked.
- **Backward Compatibility:** `user_id` is accepted as an alias for `affitor_customer_key` in metadata.

## Troubleshooting

### Conversion Not Tracked

**Check:**
- All three required fields included in Stripe metadata:
  - `affitor_click_id` (from Affitor cookie)
  - `affitor_customer_key` (your internal user/customer ID)
  - `program_id` (your Affitor program ID)
- Cookie reading function working correctly
- Cookies not blocked by browser (check browser console)
- `affitor_customer_key` matches the `customerKey` used in `signup()` during registration
- Webhook endpoint configured correctly
- Webhook events selected (`checkout.session.completed`)
- User signed up via affiliate link (customer-partner record exists)

### Wrong Partner Attributed

**Check:**
- Customer clicked partner's link before signing up
- `signup()` was called during registration with correct `customerKey`
- Customer-partner relationship exists in database
- `affitor_click_id` in metadata matches the cookie value
- `affitor_customer_key` in Stripe metadata exactly matches `customerKey` from `signup()`

### Subscription Renewals Not Tracked

**Check:**
- `subscription_data.metadata` includes all three fields:
  - `affitor_click_id`
  - `affitor_customer_key`
  - `program_id`
- `invoice.payment_succeeded` webhook event selected
- Partner within commission duration window
- Original checkout has customer-partner relationship
